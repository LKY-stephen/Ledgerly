## What v1 Is Solving

The v1 baseline is designed to work when evidence intake is intentionally sparse. Evidence only provides:

- `date`
- `amount`
- `source`
- `target`
- `description`

Everything else must come from one of two places:

- manual user input collected after the initial record is created
- deterministic derivation from stored records and supporting configuration

This keeps the first version shippable without requiring rich OCR, backend enrichment, or a large up-front evidence schema.

## v1 Data Sourcing Model

### Direct evidence-captured fields

These are the only fields assumed to arrive directly from evidence in v1:

- `date`
- `amount`
- `source`
- `target`
- `description`

They are enough to create an initial canonical record plus an evidence object.

### Manual user-input fields

These fields should be collected from the user after intake when needed:

- entity selection when the workspace has more than one reporting entity
- record classification such as `income`, `expense`, `transfer`, or `personal_spending`
- posting intent when accounting output needs a specific posting pattern
- category and tax mapping choices
- business-use ratio
- payment method or settlement context
- currency if it cannot be defaulted reliably from entity settings
- tax-year settings such as cash-method assumptions
- any tax inputs that are not directly derivable from ledger records

The v1 schema should treat these as durable user-owned fields rather than pretending they came from evidence extraction.

### Deterministic derived data

These fields and outputs should be derived after persistence:

- resolved counterparty links from `source` and `target`
- normalized sign or direction logic once the user confirms classification
- posting lines derived from a canonical record plus posting pattern
- ledger summaries, journal views, and account balances
- line-level tax projections that are safe to derive from records and manual tax inputs

Derived outputs should be reproducible from stored records and supporting configuration, not handwritten in downstream screens.

## v1 Table Design

The v1 baseline does not need the entire runtime v4 surface on day one. It needs a minimal schema that preserves a clean upgrade path while still being practical to implement.

### 1. `entities`

Purpose: one reporting owner per local book.

Minimum fields:

- `entity_id`: stable primary key
- `legal_name`: reporting name
- `entity_type`: sole proprietor / LLC / other legal form
- `base_currency`: default currency for new records
- `default_timezone`: default timezone for date handling
- `created_at`: creation timestamp

Why it exists in v1:

- every record must belong to a reporting owner
- sparse evidence does not tell the app which legal entity owns the transaction
- defaults such as currency and timezone should come from entity settings, not from evidence parsing

### 2. `counterparties`

Purpose: normalize raw `source` and `target` strings into durable business parties.

Minimum fields:

- `counterparty_id`: stable primary key
- `entity_id`: owning entity
- `display_name`: user-facing party name
- `counterparty_type`: platform, client, vendor, bank, owner, tax agency, or unknown
- `raw_reference`: optional original raw label from evidence intake
- `notes`: optional manual notes
- `created_at`: creation timestamp

Why it exists in v1:

- evidence gives `source` and `target`, but those should not remain forever as unstructured strings
- the app needs a stable place to resolve repeated parties over time
- v1 can start with light normalization and enrich later

### 3. `records`

Purpose: canonical operational ledger row for each finance event.

Minimum fields:

- `record_id`: stable primary key
- `entity_id`: owning entity
- `record_status`: lifecycle state such as `draft`, `classified`, `posted`
- `occurred_on`: canonical date from captured evidence
- `amount_cents`: canonical amount in minor units
- `currency`: transaction currency, defaulted from entity when not entered manually
- `description`: captured description
- `source_label`: raw captured `source`
- `target_label`: raw captured `target`
- `source_counterparty_id`: optional resolved source counterparty
- `target_counterparty_id`: optional resolved target counterparty
- `record_kind`: manual business meaning such as `income`, `expense`, `transfer`, `personal_spending`
- `posting_pattern`: manual or assisted posting intent used by derived views
- `category_code`: optional manual category
- `tax_category_code`: optional manual tax mapping
- `business_use_bps`: optional manual business-use allocation
- `created_at`: creation timestamp
- `updated_at`: update timestamp

Design note:

- v1 should create the `records` row immediately from the five captured fields plus safe defaults
- richer accounting meaning is layered onto the same canonical row afterward instead of delaying persistence

### 4. `record_entry_classifications`

Purpose: preserve the user's interpretation separately from the raw canonical record.

Minimum fields:

- `record_id`: one-to-one key to `records`
- `entry_mode`: `standard_receipt`, `manual`, or other input mode
- `user_classification`: user-selected interpretation such as `income`, `expense`, `transfer`, `personal_spending`
- `classification_status`: `pending`, `confirmed`, `needs_review`
- `resolver_code`: optional local heuristic result
- `resolver_note`: optional explanation for review state
- `created_at`: creation timestamp
- `updated_at`: update timestamp

Why it exists in v1:

- sparse evidence is not enough to infer bookkeeping meaning reliably
- separating interpretation from the base record keeps the system auditable and easier to revise

### 5. `tax_year_profiles`

Purpose: store manual tax context that does not belong on a single transaction row.

Minimum fields:

- `entity_id`: owning entity
- `tax_year`: target year
- `accounting_method`: cash or other supported method
- `created_at`: creation timestamp
- `updated_at`: update timestamp

Why it exists in v1:

- tax settings are not evidence facts
- v1 should not overload transaction intake with entity-year tax configuration

### 6. `evidences`

Purpose: logical evidence object for one uploaded or captured source document.

Minimum fields:

- `evidence_id`: stable primary key
- `entity_id`: owning entity
- `evidence_kind`: receipt, bank item, invoice, payout statement, note, other
- `captured_date`: captured `date` when the evidence object itself is created from intake
- `captured_amount_cents`: captured `amount`
- `captured_source`: captured `source`
- `captured_target`: captured `target`
- `captured_description`: captured `description`
- `source_system`: mobile capture / import / manual entry marker
- `created_at`: creation timestamp

Why it exists in v1:

- the app should preserve what came directly from evidence before user edits and derivations reshape the bookkeeping row

### 7. `evidence_files`

Purpose: physical file metadata for local vault storage.

Minimum fields:

- `evidence_file_id`: stable primary key
- `evidence_id`: parent evidence
- `vault_collection`: local vault collection
- `relative_path`: stored file path
- `original_file_name`: original filename
- `mime_type`: media type
- `size_bytes`: file size
- `sha256_hex`: content hash
- `captured_at`: file capture timestamp
- `is_primary`: primary file flag

Why it exists in v1:

- sparse evidence input still needs durable local file storage and attachment metadata

### 8. `record_evidence_links`

Purpose: many-to-many linkage between records and evidence objects.

Minimum fields:

- `record_id`: linked record
- `evidence_id`: linked evidence
- `link_role`: supporting, primary, imported_from, or other relationship
- `is_primary`: primary evidence flag for the record
- `notes`: optional manual notes
- `created_at`: creation timestamp

Why it exists in v1:

- one evidence item can support multiple records
- one record may rely on multiple evidence items
- v1 should not encode a brittle one-file-to-one-record assumption

## v1 View Design

v1 should still formalize derived views, even if it ships with fewer projections than runtime v4.

### 1. `record_normalized_v`

Purpose: stable normalized read surface for record review screens.

Expected outputs:

- `record_id`
- `entity_id`
- `occurred_on`
- `amount_cents`
- `currency`
- `description`
- `source_label`
- `target_label`
- `record_kind`
- `classification_status`
- `business_use_bps`

Why it matters:

- gives UI and review flows one clean read surface
- avoids repeating local normalization logic across screens

### 2. Schedule C and Schedule SE query surface

Purpose: keep the first working tax previews available without requiring the removed ledger and tax-registry layers.

Expected runtime inputs:

- `records.entity_id`
- `records.record_status`
- `records.occurred_on`
- `records.amount_cents`
- `records.currency`
- `records.record_kind`
- `records.tax_line_code`
- `records.business_use_bps`
- `tax_year_profiles.accounting_method`

Expected runtime behavior:

- Schedule C candidate reads filter directly from `records`
- Schedule C totals are aggregated in application code from those candidate rows
- Schedule SE preview is derived from the Schedule C aggregation
- if the preview still needs explicit user review or additional scope, hooks expose that requirement to the caller

Why it matters:

- preserves the first tax preview workflow the user asked to keep
- avoids forcing `accounts`, posting views, or `tax_lines_v` into the selected baseline

## v1 Feasibility Under Sparse Evidence

The sparse-input model is feasible because the system does not require full accounting truth at intake time.

The practical flow is:

1. Capture the five direct evidence fields.
2. Create a canonical record and linked evidence object locally.
3. Ask the user for only the additional fields required to classify and post the record correctly.
4. Derive normalized views and the retained tax previews from the stored record plus manual settings.

This works because v1 is records-first, not parser-first.

It avoids two common failure modes:

- blocking record creation until every accounting field is known
- pretending missing evidence data was extracted automatically when it was not

## Follow-Up Risks

- If v1 does not clearly separate captured fields from manual fields, later implementations may overcomplicate intake.
- If future documents reuse runtime v4 details without labeling them as beyond-v1, the baseline can drift upward unintentionally.
- If tax automation is expected too early, the sparse evidence model may be blamed for problems that actually come from missing manual workflow steps.
